# HARVEST.md — 从已验证的项目里提炼标准的协议

> 用途:当你打开一个「已经做得不错」的项目、想把其中的优秀实践沉淀进设计标准时,
> 让 AI 按本协议执行即可,不必每次重新交代流程。
>
> 触发方式(对 Codex 说):**「当前项目是 `<X>`,请按 `design-standard/HARVEST.md` 执行提炼。」**

## 前提
- 当前工作目录是那个「不错的项目」`<X>`。
- `design-standard/` 是它的子模块(若没有,先 `git submodule add <标准仓库地址> design-standard`)。

## 核心原则:是「提炼」,不是「搬运」
绝不把项目里的代码整段抄进标准。只提取**已被验证、且可泛化**的精华,并归入正确的层。

## 执行步骤

### 第 1 步 —— 只输出「提炼报告」,不要改任何文件
扫描本项目,列出你认为**做得好、且值得复用**的东西。每一条写明:
- **是什么**(具体到组件 / 页面结构 / 交互处理 / 视觉数值)
- **为什么好**
- **建议归入哪一层**(见下方「分层」)
- **是否与现有标准冲突**(和 `PRINCIPLES.core.md` / 现有主题对照)

### 第 2 步 —— 过滤
只有**同时满足**才可提升,否则丢弃:
1. 真的好(等我确认);
2. 可泛化(不是本项目特有的业务逻辑 / 一次性需求);
3. 不与现有 core 或现有主题冲突。

### 第 3 步 —— 等我逐条确认
我确认哪些条目可以进、进哪一层之后,你才动手。拿不准时**先问我**,默认不提升。

### 第 4 步 —— 写入并 push 子模块
把确认的条目写进 `design-standard/` 对应位置,然后 **commit + push 该子模块**,
使其它项目可通过 `git submodule update --remote` 拉取。

## 分层(提炼出来的东西放哪)
- **视觉数值**(配色 / 圆角 / 字体 / 密度)→ 对应主题的 `themes/<主题>/tokens.css`。
  - 若本项目视觉气质与现有主题**明显不同但也很好** → 复制 `themes/_template` 为新主题,用本项目真实数值填入。
  - 若气质相同 → 丰富现有主题的 token / 规则。
- **组件形态**(某控件的类型 / 状态 / 规则)→ `components/<name>.md`(按 `components/_template.md` 结构)。
- **页面信息架构**(设置页怎么拆、表格头部怎么排、modal vs drawer……)→ `patterns/<name>.md`。
- **交互反馈处理**(loading / 成功 / 失败 / 乐观更新)→ `interaction/<name>.md`。
- **一句话审美判断** → 通用进 `DECISIONS.core.md`;风格专属进 `themes/<主题>/DECISIONS.md`。

## 反模式(别做)
- ❌ 一次性预写大量规范文件——只沉淀这个项目里**真实存在且验证过**的东西。
- ❌ 把业务特有逻辑当成通用标准。
- ❌ 未经我确认就写入或 push 标准。

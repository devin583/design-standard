# 写操作反馈铁律 (mutation-feedback)

> 适用于一切「会改变数据的操作」:创建、保存、编辑、删除、提交、状态流转……
> 这是工具系统的基本盘,与视觉风格无关。**这些是 MUST,不是建议。**

## 铁律

1. **必须有 pending 状态。**
   触发写操作后,被点的控件立即进入加载/禁用态(按钮转圈或置灰),让用户知道「在处理」。

2. **禁止重复提交。**
   pending 期间该操作不可再次触发(按钮禁用 / 防抖)。提交两次订单这种事不允许发生。

3. **成功必须有可见反馈。**
   不允许「静默成功」——后台成功了但界面毫无变化。成功后必须满足下列之一:
   - 本地 UI 立即反映新状态(列表新增行、状态徽标变化等),**或**
   - 明确的成功提示(toast / 内联提示),**或**
   - 关闭弹窗并回到已更新的列表。

4. **成功后本地必须刷新或乐观更新。**
   要么重新拉取、要么乐观更新本地状态。绝不能让用户看着旧数据以为没生效。

5. **失败不能清空用户输入。**
   写操作失败时,表单里用户填的内容必须保留,只提示错误、允许重试。让用户重填是不可接受的。

6. **失败必须给出可读的错误反馈。**
   说明哪里出错、能否重试。不要只在 console 里报错而界面无感。

## 状态机(每个写操作都应覆盖)
`idle → pending → (success | error)`
- **pending**:控件禁用 + 加载指示
- **success**:可见反馈 + 本地状态更新
- **error**:保留输入 + 可读错误 + 可重试

## Loading 状态补充
- 局部操作用**局部**加载态(就在那个按钮 / 那一行),不要整页 spinner。
- 首屏 / 大块数据加载可用骨架屏(skeleton),但不要让骨架屏闪烁(<300ms 的加载就别显示)。

## Do
- 点「创建」→ 按钮转圈禁用 → 成功 → 弹窗关闭,新事项出现在列表顶部。
- 内联编辑标题 → 失焦保存 → 旁边出现短暂的「已保存」→ 失败则恢复可编辑并提示。

## Don't
- ❌ 点保存后没有任何变化,用户不知道成功没成功。
- ❌ 提交失败后整个表单被清空。
- ❌ 没禁用按钮,用户连点三次创建出三条重复数据。
- ❌ 用整页遮罩 spinner 处理一个本可局部完成的小操作。
